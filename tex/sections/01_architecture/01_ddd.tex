\subsection{领域驱动设计架构思想}

本项目采用领域驱动设计（Domain-Driven Design, DDD）来组织代码，其核心目标不是“按技术栈分目录”，而是“按业务语义建模并约束依赖方向”。
在我们的 CPU-Lab 语境中，业务语义对应为实验对象（矩阵点积与求和归约）、实验流程（配置加载、批量执行、结果落盘）以及外部交互（CLI）。

我们将系统划分为四层：\textbf{Interfaces}、\textbf{Application}、\textbf{Domain}、\textbf{Infrastructure}。
这种分层遵循依赖倒置原则（Dependency Inversion Principle, DIP）：高层策略不直接依赖底层细节，底层通过实现接口被高层使用。
这样做有三个直接收益：
\begin{itemize}
    \item \textbf{可演化性}：新增算法策略（例如 cache 优化或 superscalar 优化）时，主要变更集中在 Domain，不破坏 CLI 与流程编排。
    \item \textbf{可测试性}：Application 与 Domain 可在不依赖真实 I/O 的条件下进行单元测试（Unit Testing），降低测试耦合。
    \item \textbf{可替换性}：计时器、CSV 读写、平台信息等基础设施可替换实现而不影响领域模型。
\end{itemize}

对于本实验要求中的“高精度计时、多次重复、最小二乘近似、跨规模分析”，DDD 的价值在于把“实验科学方法”固化为稳定的领域与应用骨架，使算法实现与实验治理解耦。
最终我们可以在保持接口稳定的前提下，持续迭代算法策略与数据分析流程。
